# FeedForward — Context & Domain Language

The shared vocabulary and architectural shape of FeedForward, for humans and agents
navigating the codebase. Decisions live in [`docs/adr/`](docs/adr/); this file is the
*language* those decisions are written in.

## What it is

A web platform for **AI-assisted formative assessment**. A student submits a **draft**;
the system generates **feedback** aligned to an instructor's **rubric**; the student
iterates across multiple drafts. Built for Curtin University on Python/FastHTML/SQLite.

## Core principles (the constraints everything answers to)

- **Privacy-first** — student submission *content* is stored only until feedback is
  generated, then deleted (see ADR 002 data-lifecycle, ADR 008 submission-privacy).
  Derived artefacts that are *not* the content (scores, signals, feedback text) persist.
- **Iterative improvement** — assessment is multi-draft; progress *across* drafts is
  first-class.
- **Multi-model** — feedback is generated by several LLMs and **aggregated**, not taken
  from a single model.
- **Instructor-controlled** — instructors own rubrics, feedback tone/detail, model
  choice, and (going forward) approve feedback before students see it.

## Ubiquitous language (glossary)

| Term | Meaning |
|---|---|
| **Role** | `Student`, `Instructor`, or `Admin` (`app/models/user.py`). |
| **Course / Enrollment** | A course and a student's membership in it (`app/models/course.py`). |
| **Assignment** | A task within a course; carries spec, feedback config, and a rubric. |
| **Rubric / RubricCategory** | The grading scheme; categories carry a `weight` (`app/models/assignment.py`). |
| **Draft** | One submitted version of a student's work. Has a `version`, a `status`, and *temporary* `content`. |
| **Submission** | The act/payload of handing in a draft (`text`, `file`, or `mixed`). Text is extracted via `app/utils/file_handlers.py`. |
| **AIModel** | A configured LLM (`provider`, `model_id`, `api_config`); `owner_type` is `system` or `instructor`. |
| **ModelRun** | One execution of one model against one draft (`run_number`, `prompt`, `raw_response`). |
| **CategoryScore** | A `score` (0–100) + `confidence` for one rubric category, produced by one ModelRun. The atom of evidence. |
| **FeedbackItem** | A qualitative note from a ModelRun: `strength`, `improvement`, or `general`. |
| **Aggregation** | Combining CategoryScores across ModelRuns per category. Methods: `Average`, `Weighted Average`, `Median`, `Maximum`, `Trimmed Mean` (`AggregationMethod`). |
| **AggregatedFeedback** | The per-category aggregated result for a draft, with an instructor-review `status`. |
| **Feedback** | The instructor-approved, released feedback record (`overall_score`, `instructor_approved`). |
| **AssessmentHandler** | The per-type *declaration* seam (essay/code/math/…): declares `validate_submission`, `get_signal_sources` (which lens analysers apply to this type), `get_prompt_template`, `format_feedback`. Built from **in-process constants**, not a DB scan. Does **not** compute metrics — lens does (ADR 012 §A3, amends ADR 011). |

### New signal vocabulary (ADR 012)

| Term | Meaning |
|---|---|
| **Signal** | A deterministic metric extracted from a submission by the **lens** analyser family (e.g. `flesch_score`, `vocabulary_richness`, citation `integrity_score`). |
| **Signal source** | Which analyser produced it (`document-analyser`, `code-analyser`, `cite-sight`, …), called over HTTP. |
| **SignalRule** | A configured mapping: `signal → rubric category` via a `transform` (`band` thresholds or `linear` clamp) + `weight`. |
| **Signal Engine** | The synthetic `AIModel` (`provider="signal"`) under which signal-derived CategoryScores are recorded, so they flow through the *same* aggregation as LLM scores — **evidence alongside the LLM, not a replacement**. |

## Lifecycles

**Draft.status:** `submitted → processing → feedback_ready → archived`
(`Draft.preprocessing_status`: `pending → processing → complete | error`).

**AggregatedFeedback.status:** `pending_review → approved → released`
(human-in-the-loop; the instructor adjusts before a student sees anything).

## Layers (where logic lives)

- `app/models/` — dataclass-based tables (FastHTML ORM). The domain nouns above.
- `app/routes/{auth,admin,instructor,student}/` — HTTP handlers, grouped by role.
- `app/services/` — business logic, decoupled from routes (feedback generation,
  rubric generation, progress analysis, prompt building; **signal extraction/scoring**
  to be added here per ADR 012).
- `app/assessment/` — the assessment-type plugin system (base + registry + handlers).
- `app/utils/` — cross-cutting helpers (file extraction, formatting, auth, and the
  new `analyser_client.py` HTTP client).

## Key invariants

- **Content is ephemeral; derived data is durable.** Never make a feature depend on
  `Draft.content` still existing after `feedback_ready`.
- **CategoryScore is the single unit of scoring evidence.** New evidence sources
  (signals) join by producing CategoryScores, not by special-casing aggregation.
- **Analysis is not FeedForward's job.** Metric/NLP computation lives in the **lens**
  family; FeedForward consumes signals over HTTP and maps them to rubrics. Missing
  signals get added in the relevant lens analyser, not here.
- **Instructors gate release.** Feedback reaches a student only after approval.
